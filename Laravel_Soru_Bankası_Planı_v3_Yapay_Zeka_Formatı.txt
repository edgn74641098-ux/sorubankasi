[KULLANICI ROLLER VE YETKILERI]

Admin (1-2 kişi):
  - Tüm sistem yönetimi
  - Kullanıcı/rol yönetimi
  - Soru CRUD + versioning + rollback
  - Import yönetimi
  - Ayar yönetimi (kritik ayarlar = şifre doğrulama)
  - Bakım modu
  - Yedekleme
  - Audit log görüntüleme
  - Spot-check kuyruk (kalite kontrolü)

Editor (10-20 kişi):
  - Soru ekleme/düzenleme (kendi sorularını)
  - Kullanıcı soru onay/red (zorunlu gerekçe)
  - Kalite kuyruğu inceleme
  - Import manual_review kuyruk
  - Soru itiraz inceleme
  - Leaderboard (salt okunur)

Kullanıcı (1000+):
  - Test çözme
  - Soru gönderme (günde max 20)
  - Kendi istatistiklerini görme
  - Leaderboard görüntüleme
  - Soru itiraz + zorluk puanı (P1)

[ÜRÜN RULES (Kesin Scope)]

Test Yapısı:
  - 20 soru / 30 dakika
  - 5 şıklı çoktan seçmeli (A-E)
  - Puanlama: Doğru +5, Yanlış 0, Boş 0 (max 100 puan)
  - Test Feedback: INSTANT_FEEDBACK_LOCKED | DELAYED_FEEDBACK | NO_FEEDBACK (admin seçer)

Soru Türleri (v1):
  - Sadece 5 şıklı çoktan seçmeli
  - Soru metni: 20-4000 karakter
  - Açıklama: Zorunlu, min 20 karakter
  - Zorluk skoru: 1.0-10.0 (dinamik, Bayesian smoothing)

Soru Durumları:
  - pending: İlk gönderim
  - approved: Editör onayladı
  - rejected: Editör/Admin reddetti
  - admin_review: Kalite eşiği başarısız
  - archived: Eski soru

Test Başlangıç Modları:
  - Rastgele: Ders havuzundan rasgele 20 soru
  - Zorluk: Kullanıcı seçen aralıktan 20 soru
  - Takıldıklarım: Yanlış yapılan sorulardan öncelik, fallback algoritmasıyla rastgele tamamlama

Leaderboard Kuralları:
  - Genel: Son 30 gün, min 3 geçerli test
  - Ders Bazlı: İlgili derste son 30 gün, min 3 geçerli test
  - Tie-breaker: Puan > Yanlış Sayısı az > Bitirme zamanı erken > Hesap eski

Kullanıcı Katkı Ödülü:
  - Onaylanan her soru: +10 puan
  - Reddedilen soru: Puan yok
  - Soru geri alındığında: +10 puan kaldırılır

BÖLÜM 3: FUNCTIONAL REQUIREMENTS (Fonksiyonel Gereksinimler)
═══════════════════════════════════════════════════════════════════════════════════════════════

[TESTİN YAŞAM DÖNGÜSÜ]

1. Kullanıcı ders seçer → Test başlat modal açılır
2. Mod seçer (Rastgele / Zorluk / Takıldıklarım)
3. Sistem 20 soru oluşturur, test başlar
4. 30 dk countdown başlar (sunucu zamanı tabanlı)
5. Kullanıcı soruları cevaplar:
   - INSTANT_FEEDBACK_LOCKED: Cevap anında gösterilir, kilitlenir
   - DELAYED_FEEDBACK: Geri bildirim yoktur, sonda gösterilir
6. Süre dolunca test otomatik sonlanır veya kullanıcı bitir butonu basabilir
7. Sonuç ekranı: Puan + Detay (doğru/yanlış/boş)
8. User profile'da istatistik güncellenir

[SORU YÖNETIM AKIŞI]

Adminler / Editörler tarafından soru ekleme:
  1. Soru formu (metni, 5 şık, doğru cevap, açıklama)
  2. Validation (metni 20-4000 char, açıklama min 20 char, şıklar boş değil)
  3. Soru immediately "approved" status'unda kaydedilir (admin/editör sorularında quality check yok)

Kullanıcıdan gelen soru:
  1. Kullanıcı formu gönderir → pending status
  2. Editör panelinde "Bekleyen Sorular" kuyruğu
  3. Editör onaylarsa → Auto kalite eşik kontrolü:
     - Başarılı: approved + +10 puan → questions tablosuna kaydedilir
     - Başarısız: admin_review (admin spot-check kuyruğu)
  4. Admin inceleyip onay/red verir
  5. Editör reddetme: Zorunlu gerekçe + rejection logu

Soru Versiyonlama:
  - Her değişiklikte version kaydı oluşur
  - Admin rollback yapabilir
  - Rollback sonrası: Etkilenen testler flaglenir + notification gönderilir

Import (CSV/XLSX):
  1. Dosya yükle (max 20 MB, 1000 satır)
  2. Parse ve validation
  3. Önizleme (hata raporuyla)
  4. İdempotency: MD5 hash (skip | merge | manual_review seçim)
  5. Confirm ve kaydet

[LEADERBOARD SNAPSHOT]
  - Her 5 dakikada bir async job çalışır
  - Global ve subject-wise snapshot'lar hesaplanır
  - Eligibility check: 30 gün, min 3 test
  - Tie-breaker uygulanır

[SORU İTİRAZ VE KALITE]
  - Kullanıcı test sırasında soru İtiraz ikonuna tıklayabilir
  - Kategori: Yanlış Cevap | İfade Belirsizliği | Yazım Hatası | Diğer
  - Editör panelinde "İtiraz Kuyruğu"
  - Editör: Çöz veya Reddet (karar logu)

Kalite Kuyruğu:
  - Metrik: wrong_rate = wrong_count / (correct_count + wrong_count)
  - Eşik (30+ deneme sonrası): wrong_rate < 0.20 veya > 0.75
  - Admin dashboard'da iki kart: "Çok Kolay" ve "Çok Zor"

Monolithic Approach (v1):

  - BLL (Business Logic Layer): Service classes
  - DAL (Data Access Layer): Models + Repositories
  - Controller: Thin controllers, business logic'i service'ye ver

Layered Architecture:
  - Controller → Service → Repository → Model → Database
  - Policy/Gate: Authorization rules
  - Middleware: CSRF, Rate limit, Auth

[DATA FLOW]

Test çözüm flow:
  1. Client: POST /tests/{id}/answer (question_id, user_answer)
  2. Server: 
     - Validate: Test still active, question belongs to test
     - Save: test_items.user_answer, test_items.answered_at
     - Calculate: is_correct, awarded_points
     - Response: {is_correct, feedback (if INSTANT_FEEDBACK_LOCKED)}
  3. Test bitiminde: POST /tests/{id}/finish
     - Calculate: score, correct_count, wrong_count, blank_count
     - Update: user_subject_stats
     - Queue: Leaderboard snapshot job (async)
     - Response: {score, details}

Leaderboard flow:
  1. Async Job (every 5 min): CalculateLeaderboardSnapshot
  2. Query: SELECT users filtered by eligibility (30 days, 3 tests)
  3. Calculate: score (test bazında), rank
  4. Insert/Update: leaderboard_global_snapshots, leaderboard_subject_snapshots
  5. Client: GET /leaderboard/global
     - Return: cached snapshot (< 5 min old)
     - User's current rank: GET /leaderboard/me (real-time calc)

8. user_submitted_questions

9. tests

12. user_wrong_question_stats

13. leaderboard_global_snapshots
    - id (PK), user_id (FK), score, rank, snapshot_at

14. leaderboard_subject_snapshots
    - id (PK), user_id (FK), subject_id (FK), score, rank, snapshot_at
    - INDEXES: (subject_id, user_id), (snapshot_at DESC)

15. question_versions
    - id (PK), question_id (FK), version_no (INT), changed_by (FK),

17. settings
    - id (PK), key (UNIQUE), value_type (ENUM: string|integer|boolean|json),

18. question_reports (P1)
    - id (PK), user_id (FK), question_id (FK), category (ENUM),

19. question_difficulty_ratings (P1)
    - id (PK), user_id (FK), question_id (FK), rating (INT 1-10),

BÖLÜM 9: TEST MOTORU ALGORITMALARI (Kesin Tanımlar)
═══════════════════════════════════════════════════════════════════════════════════════════════

[MOD 1: RASTGELE TEST]
1. Ders havuzundan onaylı tüm sorular
2. 20 tanesini random select
3. Soru tekrarına izin verilir
4. Sonuç: 20 soru array

[MOD 2: ZORLUK TEST]
1. Kullanıcı range seçer (örn. 5-8)
2. Ders havuzundan difficulty_score aralığında onaylı sorular
3. 20 tanesini random select
4. Fallback: Range'den 20 bulunamazsa tüm dersten tamamla
5. Sonuç: 20 soru array

[MOD 3: TAKIILDIKLARIM - KESIN FALLBACK ALGORITMASI]

Adım 1: Öncelik Havuzu Oluştur
  - Query: user_wrong_question_stats WHERE user_id = ? AND wrong_count > 0
  - AND question_id NOT IN (son 72 saat çözülen sorular)
  - Kalan soru sayısı ≥ 20 ise Adım 3'e git

Adım 2: Kademeli Cooldown İndir
  - IF soru sayısı < 20:
    * Cooldown'u 48 saate düşür, tekrar kontrol
    * IF hala < 20: 24 saate düşür
    * IF hala < 20: Cooldown tamamen kaldır
  - Artık yanlış yapılan tüm sorular seçilebilir

Adım 3: Eksik Soruları Tamamla
  - Remaining = 20 - current_count
  - SELECT random questions FROM derste onaylı sorular LIMIT Remaining
  - Total 20'ye tamamla

Adım 4: Başarısızlık Durumu
  - IF tüm adımlar sonunda < 20:
    * ERROR: 422 mode_pool_insufficient
    * Message: "Takıldıklarım modunda yeterli soru yok"

Adım 5: Kısmi Fallback Bildirim
  - IF Adım 3'te rastgele sorular eklendiyse:
    * Kullanıcıya mesaj: "Takıldıklarım modunda yeterli soru olmadığı için [X] soru rastgele eklendi"

[ZORLUK SKORU HESAPLAMA]

Başlangıç: 5.0

Metrikler:
  - correct_count: Doğru sayısı
  - wrong_count: Yanlış sayısı
  - attempt_count = correct_count + wrong_count
  - wrong_rate = wrong_count / attempt_count (0.0 - 1.0)

Bayesian Smoothing (aşırı oynaklık önlemek):
  smoothed_wrong_rate = (wrong_count + 3) / (attempt_count + 6)

Formül:
  difficulty_score = CLAMP(1.0, 10.0, ROUND(1.0 + 9.0 * smoothed_wrong_rate, 1))

Sonuç:
  - 1.0-3.9: Easy (çoğu doğru)
  - 4.0-6.9: Medium
  - 7.0-10.0: Hard (çoğu yanlış)

Hibrit Zorluk (P1 - %70 performance + %30 user rating):
  final_difficulty = ROUND((0.7 * performance_difficulty) + (0.3 * user_rating_avg), 1)
  Koşul: Min 10 user rating. Aksi takdirde %100 performance.

[PUANLAMAYı]

Formula:
  - Doğru: +5 puan
  - Yanlış: 0 puan
  - Boş: 0 puan

Test Puanı:
  score = correct_count * 5 (max 100)

Leaderboard Sıralaması:
  ORDER BY
    1. score DESC
    2. wrong_count ASC
    3. finished_at ASC
    4. created_at ASC

[LEADERBOARD ELIGIBILITY]

Geçerli Test Tanımı:
  - status = 'finished'
  - question_count = 20
  - duration_minutes ≤ 30
  - aborted = false

Genel Leaderboard:
  - Son 30 gün içinde ≥ 3 geçerli test

Ders Bazlı Leaderboard:
  - İlgili derste son 30 gün içinde ≥ 3 geçerli test

Query Builder:

[RISK REGISTER]

| Risk | Severity | Probability | Mitigation |
|------|----------|-------------|-----------|
| Database performance degrade (300 concurrent) | High | Medium | Index optimization, connection pool tuning, load test staging |
| Rollback race condition (soru değiştirilirken, test çözülüyor) | High | Low | Flagging system, atomic transactions, audit log |
| Import idempotency çalışmıyor (tekrar yükleme sorun) | Medium | Medium | MD5 hash + comprehensive testing |
| Leaderboard snapshot job crashes | Medium | Medium | Error handling, retry logic, email alert |
| Test feedback mode çelişkisinden kullanıcı confusion | Medium | High | Clear UI, documentation, in-app tutorial |
| Takıldıklarım fallback algoritması bug | High | Medium | Pseudocode review, comprehensive unit tests, edge case testing |
| CSRF attack penetration | High | Low | Token verification, SameSite cookie, security test |
| XSS injection via soru metni | High | Low | Sanitization, Blade escaping, security test |
| Rate limit bypass | Medium | Medium | Redis-backed rate limit, comprehensive testing |
| Email verification bypass | High | Low | Email verification zorunlu, 403 check |
| Admin password reset abuse | Medium | Medium | Secure token generation, expiration, audit log |
| Data loss (backup failure) | Critical | Low | Backup testing, verification, multiple backups |
| Third-party API failure (email service) | Medium | Medium | Fallback mechanism, retry logic |
| Hosting cPanel limitation (cron, queue) | Medium | High | Database queue (compatible), capability check, fallback |
| Developer error (SQL, wrong logic) | High | Medium | Code review, testing, static analysis |

[MITIGATION STRATEGIES]

Critical Risks:
  1. Database performance:
     → Index from day 1
     → Staging load test (300 concurrent)
     → Connection pool tuning
     → N+1 prevention (eager loading)

  2. Rollback race condition:
     → Version conflict detection (409 error)
     → Flagging system (test_items.rollback_applied)
     → Atomic transaction
     → Audit log + notification

  3. Data loss:
     → Daily automated backups
     → Backup verification (staging restore test)
     → Multiple backup copies
     → Retention policy (30+ backups)

High Severity Risks:
  4. Takıldıklarım algorithm:
     → Pseudocode review + document
     → Unit test (6+ scenarios)
     → Staging test with real data
     → Edge case handling (< 20 sorya ulaşılamadığında)

  5. CSRF/XSS/Rate limit:
     → Security test suite
     → OWASP guidelines review
     → Manual penetration test
     → Production monitoring

  6. Hosting cPanel limitations:
     → Capability check (cron, queue support)
     → Fallback strategy (manual backup, database queue)
     → Hosting upgrade path (v1.1+)

[CONTINGENCY PLANS]

If Database Performance Degrades:
  1. Enable query slow log
  2. Analyze slow queries
  3. Add missing indexes
  4. Increase connection pool if available
  5. Consider database read replica (v1.1+)
  6. Revert to previous version if necessary

If Leaderboard Snapshot Fails:
  1. Alert team via email
  2. Retry failed job
  3. Skip snapshot for 5 min
  4. Use last valid snapshot (fallback)
  5. Investigate error in logs
  6. Manual recalculation if needed

If Import Idempotency Fails:
  1. Rollback transaction
  2. Log error with batch ID
  3. Notify admin
  4. Manual review required
  5. Prevent duplicate entries

If Backup Fails:
  1. Alert admin
  2. Retry next scheduled time
  3. Log failure
  4. Manual backup required
  5. Verify previous backup integrity

[SUCCESS METRICS & MONITORING]

Uptime: Target 99.5%
- Monitor: Application health check (every 1 min)
- Alert if: Downtime > 15 min cumulative

Response Time: P95 < 500ms
- Monitor: Application logs + APM (if available)
- Alert if: P95 > 1 second for 5 min consecutive

Error Rate: < 1%
- Monitor: Exception log + Sentry
- Alert if: Error rate > 2% for 5 min

Test Quality: Pass rate > 95%
- Monitor: CI/CD test results
- Alert if: < 90%

- Monitor: Periodic surveys
- Alert if: < 40

═══════════════════════════════════════════════════════════════════════════════════════════════

[v1.1 BACKLOG (4-6 hafta sonra)]

P1 Features (High Priority):
  - Soru itiraz akışı (P1)
  - Soru zorluk puanlaması + hibrit zorluk (P1)
  - Medya destekli soru (görsel, ses)
  - Blueprint test şablonları
  - Kullanıcı profil gizliliği toggle
  - Balanced zorluk dağılımı option

Infrastructure:
  - Redis Queue (Database queue'den migration)
  - Redis cache optimization
  - Sentry error tracking integration
  - Multi-server scaling (load balancer)
  - MySQL replication (master-slave)

[v1.2+ VISION (6+ ay)]

Gamification:
  - Daily goals / streak system
  - Achievement badges
  - Weekly leaderboard
  - Level system (beginner → expert)

Social Features:
  - Kullanıcı yorum sistemi
  - Likes / reactions on comments
  - Düello modu (head-to-head test)
  - Study groups / collaboration

Analytics:
  - Advanced reporting (admin için)
  - Student progress tracking (öğretmen için)
  - Test performance analysis
  - Common mistakes report

[TECHNICAL DEBT & IMPROVEMENTS]

Current (v1) Limitations:
  - Single-server deployment
  - File cache only (Redis recommended)
  - Database queue (Redis recommended)
  - No error tracking
  - Basic monitoring
  - No API rate limiting

Future Improvements:
  - Full-text search (Elasticsearch)
  - Real-time notifications (WebSocket)
  - Mobile app (React Native / Flutter)
  - AI-powered question generation
  - Smart recommendation engine
